                              변경관리규정
================================================================================

제1장 총칙

제1조 (목적)
본 규정은 IT 시스템 및 서비스에 대한 변경을 체계적으로 관리하여
서비스 안정성을 유지하고 변경으로 인한 장애를 최소화함을 목적으로 한다.

제2조 (적용범위)
① 본 규정은 운영 환경에 대한 모든 변경에 적용된다.
② 변경 대상:
   - 서버, 네트워크, 스토리지 등 인프라
   - 운영체제, 미들웨어, 데이터베이스
   - 응용 프로그램 (배포, 패치)
   - 보안 설정, 정책
   - 구성 정보 (Configuration)

③ 제외 대상:
   - 개발/테스트 환경 (별도 관리)
   - 단순 데이터 입력/수정 (업무 처리)

제3조 (용어의 정의)
① "변경"이란 IT 자원의 추가, 수정, 삭제를 말한다.
② "변경요청(RFC)"이란 Request for Change로 변경 신청서를 말한다.
③ "변경자문위원회(CAB)"란 Change Advisory Board로 변경 승인 기구를 말한다.
④ "변경윈도우"란 변경 작업이 허용되는 시간대를 말한다.
⑤ "롤백"이란 변경 실패 시 원래 상태로 복원하는 것을 말한다.

--------------------------------------------------------------------------------

제2장 변경 분류

제4조 (변경 유형)
| 유형 | 설명 | 리스크 | 승인 |
|------|------|--------|------|
| 긴급변경 | 장애 복구, 보안 긴급패치 | 높음 | 사후 승인 |
| 표준변경 | 사전 승인된 정형화된 변경 | 낮음 | 자동 승인 |
| 일반변경 | 기능 추가, 설정 변경 등 | 중간 | CAB 승인 |
| 대규모변경 | 시스템 교체, 아키텍처 변경 | 매우 높음 | CIO 승인 |

제5조 (표준변경 목록)
사전 승인된 표준변경 예시:
① 정기 패치 적용 (월간 정기)
② 사용자 계정 생성/삭제
③ 디스크 용량 증설 (기존 서버)
④ 백업 스케줄 변경
⑤ 로그 정리
⑥ 정기 재기동 (주간 점검 시)

제6조 (변경 리스크 평가)
| 평가 항목 | 고위험 | 중위험 | 저위험 |
|-----------|--------|--------|--------|
| 영향 시스템 | 핵심시스템 | 중요시스템 | 일반시스템 |
| 영향 범위 | 전사/다수 부서 | 특정 부서 | 소수 사용자 |
| 작업 복잡도 | 복잡, 다단계 | 보통 | 단순 |
| 롤백 가능성 | 어려움 | 보통 | 용이 |
| 서비스 중단 | 필요 | 부분 중단 | 불필요 |

제3장 변경자문위원회(CAB)

제7조 (CAB 구성)
① 상임 위원:
   - IT본부장 (위원장)
   - 인프라운영팀장
   - 개발팀장
   - IT보안팀장

② 필요시 참여:
   - 해당 시스템 담당자
   - 현업 부서 담당자
   - 협력업체 담당자

제8조 (CAB 운영)
① 정기 CAB: 매주 수요일 14:00
② 긴급 CAB: 수시 (고위험 변경, 긴급 변경)
③ 심의 내용:
   - 변경 필요성 검토
   - 리스크 평가
   - 작업 계획 적정성
   - 롤백 계획 확인
   - 변경윈도우 확정

④ 의결: 출석 위원 과반수 찬성

제9조 (CAB 결정)
① 승인: 변경 진행 허용
② 조건부 승인: 보완 후 재심의 없이 진행
③ 반려: 계획 보완 후 재심의
④ 보류: 추가 검토 필요

--------------------------------------------------------------------------------

제4장 변경 절차

제10조 (변경요청, RFC)
① 변경요청서 필수 항목:
   - 변경 제목 및 설명
   - 변경 사유 및 효과
   - 변경 대상 시스템
   - 변경 유형 및 리스크
   - 작업 계획 (일정, 담당자)
   - 테스트 계획
   - 롤백 계획
   - 영향 분석

② 제출 시한:
   - 일반변경: CAB 3영업일 전
   - 대규모변경: CAB 7영업일 전

제11조 (변경 검토)
① 1차 검토: IT기획팀
   - 형식 요건 확인
   - 유사 변경 이력 확인
   - 변경 충돌 확인 (동일 시스템 타 변경)

② 2차 검토: 해당 시스템 담당자
   - 기술적 타당성
   - 리스크 평가
   - 작업 계획 적정성

③ 보안 검토: IT보안팀 (해당 시)
   - 보안 영향 평가
   - 보안 요구사항 확인

제12조 (변경 승인)
| 변경 유형 | 승인권자 |
|-----------|----------|
| 표준변경 | 팀장 (자동 승인) |
| 일반변경 (저위험) | 팀장 |
| 일반변경 (중위험) | CAB |
| 일반변경 (고위험) | CAB + IT본부장 |
| 대규모변경 | CIO |
| 긴급변경 | IT본부장 (사후 CAB 보고) |

제13조 (변경 실행)
① 사전 준비:
   - 작업 체크리스트 확인
   - 롤백 환경 준비
   - 관련자 통보

② 작업 실행:
   - 변경윈도우 내 수행
   - 작업 로그 기록
   - 단계별 확인

③ 작업 완료:
   - 정상 동작 확인
   - 테스트 수행
   - 모니터링 강화

제14조 (변경 검증)
① 기능 테스트: 변경 기능 정상 동작 확인
② 회귀 테스트: 기존 기능 영향 확인
③ 성능 테스트: 성능 저하 여부 확인
④ 보안 테스트: 취약점 발생 여부 확인 (해당 시)

제15조 (변경 종료)
① 변경완료 보고서 작성
② 포함 내용:
   - 작업 결과 (성공/실패/부분 성공)
   - 작업 일시 및 담당자
   - 발생 이슈 및 조치
   - 후속 조치 필요 사항

③ 변경 이력 등록 (CMDB 갱신)

--------------------------------------------------------------------------------

제5장 변경윈도우

제16조 (변경윈도우 정의)
① 표준 변경윈도우:
   - 매주 일요일 02:00 ~ 06:00
   - 격주 토요일 22:00 ~ 익일 06:00

② 긴급 변경윈도우:
   - 장애 복구, 긴급 보안패치: 즉시
   - IT본부장 승인 필요

③ 금지 기간 (Freeze Period):
   - 결산 기간: 분기말 3영업일
   - 대규모 이벤트: 별도 지정
   - 연말연시: 12/24 ~ 1/3

제17조 (서비스 중단)
① 서비스 중단 최소화 원칙
② 중단 필요 시:
   - 사전 공지 (최소 3일 전)
   - 중단 시간 최소화
   - 고객 영향 시 별도 승인

③ 서비스 중단 공지:
   - 내부: 사내 포털
   - 외부: 홈페이지, 앱, SMS

--------------------------------------------------------------------------------

제6장 긴급변경

제18조 (긴급변경 정의)
긴급변경은 다음의 경우에 한한다:
① 서비스 장애 복구
② 긴급 보안 패치 (제로데이 취약점 등)
③ 법규 준수 긴급 대응
④ 기타 경영상 긴급 필요

제19조 (긴급변경 절차)
① 긴급변경 신청:
   - 전화 또는 메신저로 긴급 요청
   - IT본부장 또는 당직 관리자 승인

② 작업 실행:
   - 즉시 실행
   - 가능한 범위 내 롤백 준비
   - 작업 내용 실시간 기록

③ 사후 처리:
   - 24시간 내 긴급변경 보고서 제출
   - 차주 CAB에 사후 보고
   - 필요시 정규 변경으로 재작업

제20조 (긴급변경 남용 금지)
① 긴급변경은 진정한 긴급 상황에만 적용
② 긴급변경 비율 목표: 전체 변경의 5% 이하
③ 과다 긴급변경 발생 시 원인 분석

--------------------------------------------------------------------------------

제7장 롤백 관리

제21조 (롤백 계획 필수)
① 모든 변경은 롤백 계획 수립 필수
② 롤백 계획 포함 사항:
   - 롤백 판단 기준 (어떤 경우 롤백)
   - 롤백 절차
   - 롤백 소요 시간
   - 롤백 담당자

제22조 (롤백 실행)
① 롤백 판단:
   - 변경 후 서비스 장애 발생
   - 성능 심각 저하
   - 데이터 오류 발견
   - 보안 취약점 발생

② 롤백 승인:
   - 현장 판단: 작업 책임자
   - 상황 보고: IT본부장 (롤백 후 1시간 내)

③ 롤백 후 조치:
   - 원인 분석
   - 재작업 계획 수립
   - CAB 보고

--------------------------------------------------------------------------------

제8장 변경 이력 관리

제23조 (변경 이력 기록)
① 기록 항목:
   - 변경 번호 (고유 ID)
   - 변경 일시
   - 변경 유형
   - 대상 시스템
   - 변경 내용
   - 작업자
   - 승인자
   - 결과 (성공/실패/롤백)

② 기록 방법: 변경관리시스템 (ITSM)
③ 보관 기간: 5년

제24조 (CMDB 갱신)
① 변경 완료 후 CMDB(구성관리DB) 갱신 필수
② 갱신 항목:
   - 하드웨어 정보
   - 소프트웨어 버전
   - 구성 정보
   - 관계 정보

③ 갱신 기한: 변경 완료 후 1영업일

--------------------------------------------------------------------------------

제9장 성과 측정

제25조 (KPI)
| 지표 | 목표 | 측정 주기 |
|------|------|-----------|
| 변경 성공률 | 98% 이상 | 월간 |
| 긴급변경 비율 | 5% 이하 | 월간 |
| 변경 기인 장애 | 월 1건 이하 | 월간 |
| CAB 승인 소요시간 | 3일 이내 | 월간 |
| 롤백 비율 | 2% 이하 | 월간 |

제26조 (변경 분석)
① 월간 변경 현황 분석
② 실패 변경 원인 분석
③ 개선 대책 수립
④ CIO 월례 보고

--------------------------------------------------------------------------------

제10장 보칙

제27조 (위반 시 제재)
① 미승인 변경 실행: 경고 ~ 정직
② 긴급변경 남용: 경고
③ 변경 기인 장애 (과실): 경고 ~ 감봉
④ 롤백 계획 미수립: 경고
⑤ 변경 기록 누락: 경고

제28조 (규정 개정)
① 본 규정은 연 1회 검토하여 필요시 개정한다.
② 규정 개정은 CIO 승인으로 확정된다.

부칙
본 규정은 2024년 1월 1일부터 시행한다.
